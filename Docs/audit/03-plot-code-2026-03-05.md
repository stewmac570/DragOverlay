## PM1 — `FitToData()` calls `Refresh()` before `AutoScale()` ⚠️ HIGH

**File:** `PlotManager.cs` lines 227–231

```csharp
public void FitToData()
{
    _plot.Refresh();               // ← renders with OLD axis limits
    _plot.Plot.Axes.AutoScale();   // ← changes limits, but doesn't re-render
}
```

**What ScottPlot v5 expects:** `AutoScale()` mutates the axis limits in memory. `Refresh()` triggers a render pass using whatever limits are currently set. To auto-fit and see the result, the order must be `AutoScale()` then `Refresh()`. The current code renders the pre-zoom state, changes the limits, then returns — the user sees no change.

**Consequence:** The "Fit to Data" button (if one exists) or any caller of `FitToData()` appears to do nothing.

**Fix:** Swap the two lines.

```csharp
public void FitToData()
{
    _plot.Plot.Axes.AutoScale();
    _plot.Refresh();
}
```

## PM2 — O(n log n) sort on every scatter on every mouse pixel ⚠️ HIGH

**File:** `PlotManager.cs` line 800 inside `FormsPlot_MouseMove`

```csharp
var nearest = pts.OrderBy(p => Math.Abs(p.X - mouseCoord.X)).First();
```

This is called for every `Scatter` on every mouse move event. With 6 runs and 10 channels each:
- Up to ~18 visible scatters (Castle: 10 channels × 3 runs; RaceBox: 2 channels × 3 runs)
- Each Castle scatter has ~400–600 points
- Each `OrderBy` is O(n log n) — sorting 500 points = ~4,500 comparisons

**Cost per mouse pixel:** 18 scatters × 4,500 comparisons = **~81,000 comparisons**, repeated at whatever frequency WinForms fires `MouseMove`.

**Correct approach:** Castle X values are monotonically increasing (50ms fixed sample, no reversals). A single O(n) linear scan — or O(log n) binary search on the sorted array — finds the nearest point without sorting. Binary search against a pre-sorted `xs` array (already built in `PlotRuns`) would be ~9 comparisons instead of 4,500.

**The fix:** Replace `OrderBy(...).First()` with a direct linear scan that also captures the index in one pass (eliminating PM3 as well).

## PM3 — Two-pass nearest-point lookup in `FormsPlot_MouseMove` ⚠️ HIGH

**File:** `PlotManager.cs` lines 800–809

```csharp
// Pass 1: sort entire list to find the nearest point's X value   O(n log n)
var nearest = pts.OrderBy(p => Math.Abs(p.X - mouseCoord.X)).First();

// Pass 2: linear scan to find the same point's index             O(n)
int index = -1;
for (int j = 0; j < pts.Count; j++)
{
    if (Math.Abs(pts[j].X - nearest.X) < 1e-12)
    { index = j; break; }
}
```

Two separate passes to find one thing. Both passes could be replaced by a single O(n) scan (or O(log n) binary search) that tracks both the nearest point AND its index simultaneously.

**Additional fragility:** The second pass compares floating-point values with `< 1e-12`. If the X values come from different computations (e.g., `pts[j].X` and `nearest.X` were computed separately), floating-point precision could prevent the match, leaving `index = -1` and silently returning no hover value even though a nearest point was found.

## PM4 — `_plot.Refresh()` called on every mouse pixel ⚠️ HIGH

**File:** `PlotManager.cs` line 822

```csharp
private void FormsPlot_MouseMove(object sender, MouseEventArgs e)
{
    // ... all the per-scatter work ...
    CursorMoved?.Invoke(valuesAtCursor);
    _plot.Refresh();   // ← full re-render on every mouse move
}
```

`_plot.Refresh()` triggers a full ScottPlot render pass: all plottables, all axes, grid lines, cursor line, labels. On a typical display, `MouseMove` fires at ~60–200 events/second. With 6 runs loaded, this is 60–200 full renders per second in addition to all the O(n log n) work in PM2/PM3.

**ScottPlot v5 approach:** The cursor line (`_cursor.X = mouseCoord.X`) requires a refresh to show, but the render can be deferred. A common pattern is:
- Throttle to at most 60fps using a timer or `Stopwatch.ElapsedMilliseconds` guard
- Or use ScottPlot's built-in render throttling if available in v5.0.8

**Consequence:** With 6 runs loaded (~18 scatters × ~600 points), hover becomes visibly sluggish. Noted in architecture audit as "break point: 6 runs loaded."

## PM5 — Near-zero Y filter silently gaps the G-Force trace ⚠️ HIGH

**File:** `PlotManager.cs` line 669

```csharp
var good = rbTyped.Where(p => Math.Abs(p.Y) > 0.01).ToList();
if (good.Count < 2) continue;
```

This filter was presumably added to exclude zero-padded or invalid RaceBox points. However:

1. **G-Force X (lateral/longitudinal acceleration) legitimately passes through zero** during a drag pass — the car decelerates and the G-Force crosses zero at peak speed. Filtering `|Y| <= 0.01` creates a **gap in the G-Force trace** at every zero-crossing. The line is rendered as two separate segments with a hole where the real data is.

2. **Speed is always positive during a drag pass**, so the filter is harmless for the speed channel. But it may also drop the first few points if the car hasn't reached 0.01 mph yet (implausible) or the last few points if data tails off to zero.

3. The filter is applied after `rbTyped.OfType<ModelPoint>()` — if the `OfType` already removes null/wrong-type entries, the additional near-zero filter is redundant for invalid data and harmful for valid G-Force data.

**Fix:** Separate the "filter noise" intent from the "filter zero G-Force" effect. For G-Force, drop only the filter (or use a much smaller threshold like 1e-9 to catch only truly zero/NaN values). For speed, the filter is arguably fine but could be documented.

## PM6 — Y-axis switch expression has no default arm ⚠️ HIGH

**File:** `PlotManager.cs` lines 339–351 (also flagged as C3 in audit 01)

```csharp
s.Axes.YAxis = channelLabel switch
{
    "RPM"          => rpmAxis,
    "Throttle %"   => throttleAxis,
    "Voltage"      => voltageAxis,
    "Current"      => currentAxis,
    "Ripple"       => rippleAxis,
    "PowerOut"     => powerAxis,
    "ESC Temp"     => escTempAxis,
    "MotorTemp"    => motorTempAxis,
    "MotorTiming"  => motorTimingAxis,
    "Acceleration" => accelAxis,
    // ← no _ => default arm
};
```

A C# switch expression with no default arm throws `SwitchExpressionException` (a subclass of `InvalidOperationException`) at runtime for any non-matching input. This is not caught anywhere in the call chain.

**Triggers:**
- The legacy `"Throttle"` channel (ms-based, commented out in `ChannelColorMap` but still present in some Castle log files)
- Any future channel added to `GetChannelsWithRaw` before being added to this switch
- Case mismatch (e.g., `"ESC temp"` vs `"ESC Temp"`)

**Fix:** Add `_ => throttleAxis` as a safe default fallback.

## PM7 — Constructor calls `ReapplyAxisLocks()` before axes exist ⚠️ MEDIUM

**File:** `PlotManager.cs` lines 138–141

```csharp
// 🔒 make Y-locks stick straight away
ReapplyAxisLocks();

_plot.Refresh();
```

At this point in the constructor, `throttleAxis`, `rpmAxis`, etc. are all `null`. `ReapplyAxisLocks()` calls `ApplyAxisLocksForActiveProfile()`, which calls:

```csharp
_plot.Plot.Axes.Rules.Add(new LockedVertical(throttleAxis, thr.Min, thr.Max));
// throttleAxis is null here
```

`LockedVertical(null, ...)` is called 10 times (once per Castle axis). Whether this throws or silently stores null rules depends on ScottPlot internals. The comment "make Y-locks stick straight away" suggests this was added reactively to fix a drift bug, but calling it before `SetupAllAxes()` (which creates the axes) means the rules reference null axis objects.

**Actual effect in practice:** `SetupAllAxes()` removes and re-adds all rules on every `PlotRuns()` call, so the null rules added in the constructor are cleared before they could cause harm. But this is accidental safety — the null rules in the constructor are a latent bug that survives only because `PlotRuns()` always follows.

**Fix:** Move the constructor's `ReapplyAxisLocks()` call to after `SetupAllAxes()` (axes must be created before rules are applied), or remove it entirely since `PlotRuns()` will apply rules before the first render.

## PM8 — Axis locks applied twice per full replot ⚠️ MEDIUM

**File:** `PlotManager.cs` — interaction between `SetupAllAxes()` (called from `PlotRuns()` at line 306) and `ReapplyAxisLocks()` (called from `PlotRuns()` at line 408)

`PlotRuns()` execution order:
1. Line 289: `_plot.Plot.Axes.Rules.Clear()` — all rules removed
2. Line 306: `SetupAllAxes()` — creates axes (if null), then calls `ApplyAxisLocksForActiveProfile()` → adds 12 rules
3. Lines 306 → `SetupAllAxes` line 523: Also adds RaceBox locks (2 more rules) → 14 rules total
4. Line 408: `ReapplyAxisLocks()` — removes all non-RaceBox `LockedVertical` rules (10 Castle rules), then calls `ApplyAxisLocksForActiveProfile()` again → adds them back

Net effect is correct, but 10 Castle axis lock rules are added, removed, and added again within the same `PlotRuns()` call. The second application overwrites the first with identical values. This is dead work on every replot.

**Root cause:** `SetupAllAxes()` and `ReapplyAxisLocks()` / `ApplyAxisLocksForActiveProfile()` all manage the same axis rules but from different angles. Three methods with overlapping responsibility, no single source of truth.

## PM9 — `GetChannelsWithRaw` always allocates two identical arrays, discards one ⚠️ MEDIUM

**File:** `PlotManager.cs` lines 827–841

```csharp
private IEnumerable<(string Label, double[] RawYs, double[] ScaledYs)> GetChannelsWithRaw(RunData run)
{
    double[] GetRaw(Func<ModelPoint, double> sel) =>
        run.DataPoints.Select(sel).ToArray();

    yield return ("RPM",        GetRaw(dp => dp.Speed),         GetRaw(dp => dp.Speed));
    yield return ("Throttle %", GetRaw(dp => dp.ThrottlePercent), GetRaw(dp => dp.ThrottlePercent));
    // ... same pattern for all 10 channels
}
```

Every channel calls `GetRaw` twice with the same selector. `ScaledYs` is always identical to `RawYs`. At the call site in `PlotRuns`:

```csharp
foreach (var (channelLabel, rawYs, _) in GetChannelsWithRaw(run))
```

The third element is explicitly discarded (`_`). The `ScaledYs` array is allocated and immediately abandoned.

**Cost:** For each `PlotRuns()` call with Castle runs loaded — 10 channels × 2 arrays × `run.DataPoints.Count` doubles. For a 600-point run, this is 10 × 2 × 600 × 8 bytes = **96 KB of allocations per Castle run, of which 48 KB is immediately garbage.** With 3 Castle runs: 144 KB of pure waste per replot.

**Fix:** Remove the `ScaledYs` element from the tuple (it was never used differently). Return only `(Label, RawYs)`.

## PM10 — `GetTimeRange` ignores `"RaceBox G-Force X"` ⚠️ MEDIUM

**File:** `PlotManager.cs` lines 949–956

```csharp
else  // RaceBox run
```

Only `"RaceBox Speed"` is consulted for the RaceBox time range. `"RaceBox G-Force X"` is not checked.

## PM12 — `SetSpeedMode` is not self-contained ⚠️ MEDIUM

**File:** `PlotManager.cs` lines 106–112

```csharp
public void SetSpeedMode(bool isSpeedRun)
{
    _activeProfile = isSpeedRun ? ScaleProfile.SpeedRun : ScaleProfile.Drag;
    Logger.Log($"📐 Scale Profile → ...");
    //ReapplyAxisLocks();   ← intentionally commented out
    //_plot.Refresh();      ← intentionally commented out

```

`SetSpeedMode` only flips an enum. The profile change has no visible effect until a full replot is triggered externally. If a caller calls `SetSpeedMode(true)` and doesn't follow up with `PlotAllRuns()` or `SetupAllAxes()` + `RefreshPlot()`, the UI silently shows the wrong scale profile.

This coupling is documented in CLAUDE.md as a pitfall, but the code itself gives no indication. The commented-out lines hint at a past attempt to make it self-contained that was reverted for unknown reasons.

**Risk:** Any new caller of `SetSpeedMode` will behave incorrectly unless they know to trigger a replot. The method signature implies it should be a complete operation.

## PM13 — Floating-point drift in tick generation; 660+ allocations per replot ⚠️ MEDIUM

**File:** `PlotManager.cs` lines 909–913

```csharp
for (double pos = minX; pos <= maxX; pos += 0.05)
    manual.AddMajor(pos, pos.ToString("0.00"));
for (double pos = minX + 0.005; pos <= maxX; pos += 0.005)
    manual.AddMinor(pos);
```

**Floating-point drift:** `pos += 0.05` is a floating-point addition. Over 60 iterations (3-second window), the accumulated error can shift tick positions by a few ULP (units in the last place). For example, a tick intended at `0.50` might land at `0.4999999999999998`, which `.ToString("0.00")` would display correctly but which the tick renderer might align slightly off the grid lines. This is visually imperceptible but technically incorrect.

**Correct pattern:** `pos = minX + i * 0.05` (one multiplication instead of accumulated additions).

**Allocation count:** For a typical 3-second drag pass (-0.5s to 2.5s = 3.0s span):
- Major ticks: 3.0 / 0.05 = **60 ticks** + label string per tick
- Minor ticks: 3.0 / 0.005 = **600 ticks**
- Total: **660 `AddMajor`/`AddMinor` calls** per `PlotRuns()` invocation, plus 60 string allocations

This ternary assigns any non-Speed RaceBox channel to `raceBoxGxAxis`. It works for the current two channels but silently assigns a future third RaceBox channel (e.g., G-Force Y, GPS altitude, heading) to the G-Force axis, where its scale range (-5 to 7) would likely be wrong.

## Positive Observations

Before recommendations: several things in `PlotManager` are done well.

- **ScottPlot completely encapsulated.** `MainForm` never touches `_plot` directly — all ScottPlot calls go through `PlotManager`. A ScottPlot version upgrade would only require changes in this one file.
- **`AxisRules` pattern is correct.** Axis locks are consistently cleared and re-applied, not accumulated. The `if (rule is LockedVertical lv && ...)` filter is the right way to selectively remove rules.
- **Axis creation guards work.** `if (throttleAxis is null)` prevents duplicate axis creation across multiple `PlotRuns()` calls.
- **`SetRun(slot, null)` cleans up correctly.** The scatter cleanup loop (lines 241–264) removes all associated plottables, dictionaries, and split lines for a cleared slot. This avoids dangling references.
- **`_rawYMap` pattern for hover values is clean.** Storing the raw Y array alongside each scatter and looking it up at hover time avoids re-querying the model during hover.
- **Split line system is well-isolated.** `_splitLinesBySlot` and `_splitLabelsBySlot` correctly track per-slot split line plottables.

## Architecture Note — Rule Management Complexity

The three-method axis lock system (`SetupAllAxes` → `ApplyAxisLocksForActiveProfile`, `ReapplyAxisLocks` → `ApplyAxisLocksForActiveProfile`) is the single most confusing structural pattern in `PlotManager`. Every method that touches axis rules has to know which rules to preserve and which to rebuild. The current split works but is brittle: changing a scale value requires knowing all three callsites.

A cleaner single-method approach would be:

```
private void ApplyAllAxisRules()
{
    _plot.Plot.Axes.Rules.Clear();           // always rebuild from scratch
    // RaceBox locks (fixed)
    // Castle locks (from active profile)
    // Split label lock
}
```

Called at the end of `PlotRuns()` and `SetChannelScale()`, never from the constructor. This eliminates PM7, PM8, and the confusion entirely.
