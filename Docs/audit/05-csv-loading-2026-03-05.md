## CsvLoader.cs — Failure Map

### CL1 — Empty file throws at `ReadHeader()` ⚠️ HIGH

**Lines:** 110–113

```csharp
csv.Read();         // returns false — nothing to read
csv.ReadHeader();   // called on a reader with no current record
Logger.Log("[DEBUG] Available CSV Headers: " + string.Join(", ", csv.HeaderRecord));
```

**Input:** File with no content at all, or a file consisting only of `#` metadata lines and nothing else.

After the metadata skip loop, `csv.Read()` finds no rows and returns `false`. `csv.ReadHeader()` is then called without a valid current record. In CsvHelper v33, this throws `CsvHelperException: "No header record was found."` or leaves `csv.HeaderRecord` null. If `HeaderRecord` is null, `string.Join(", ", null)` throws `ArgumentNullException`.

**Propagation:** Both exceptions propagate to the Castle load handlers (`LoadRun1/2/3Button_Click`), which DO wrap the `await Task.Run(() => loader.Load(...))` in try/catch. The exception is caught, a MessageBox is shown, and the UI resets. **This is the one place in the codebase where the calling pattern is correct.** The exception is handled.

**However:** The error message shown to the user comes from CsvHelper internals ("No header record was found"), not from the application. A user loading a file by mistake would see a cryptic error.

### CL2 — Missing "Power-Out" column silently returns empty data ⚠️ HIGH

**Lines:** 130–148

```csharp
string rawPowerOut = csv.GetField<string>("Power-Out");
// With MissingFieldFound = null, this returns null if "Power-Out" doesn't exist
double powerOut = 0.0;
if (!string.IsNullOrWhiteSpace(rawPowerOut)) { ... }

if (trimForDrag)
{
    if (!launchPointFound && powerOut >= 5.0)
        launchPointFound = true;
    if (!launchPointFound)
    {
        rowIndex++;
        continue;   // ← every row skipped when powerOut is always 0
    }
}
```

**Input:** A Castle CSV where the power column is named `"Power Out"`, `"PowerOut"`, `"Power (W)"` instead of `"Power-Out"`. Also triggered by any non-Castle CSV file accidentally opened.

Because `MissingFieldFound = null`, CsvHelper returns null for the unknown column. `powerOut` stays 0.0 for every row. `launchPointFound` never becomes true. Every data row is skipped via `continue`. The parsing loop finishes with `runData.DataPoints.Count == 0`.

After the loop, the trim check is:
```csharp
if (runData.DataPoints.Count > 100 && ...)
```
This is false — AutoTrim is skipped with a silent log entry. The method **returns an empty `RunData` with no user notification**. The UI button label resets to show the filename, implying the file loaded successfully, but nothing plots.

**There is no MessageBox shown for this failure path.** The user sees their file appear to load with no visible data, and no explanation.

### CL3 — `double.TryParse` without `CultureInfo` zeros all channels on non-EN locales ⚠️ HIGH

**Lines:** 81–95, 135, 159

Three separate call sites:

```csharp
// Metadata section (lines 81–95): throttle calibration values
double.TryParse(meta.Split(':').Last()..., out double v)

// Power-Out parsing (line 135):
double.TryParse(rawPowerOut, out powerOut)

// GetDouble() — used for every channel (line 159):
return double.TryParse(raw, out double result) ? result : 0.0;
```

None of these pass `CultureInfo.InvariantCulture`. Castle CSV files always use `.` as the decimal separator. On machines with a German/French/Spanish/etc. locale, `double.TryParse("8.4", out v)` returns `false` (`.` is the thousands separator in those locales, not valid mid-number without a following thousands digit). The result is silently `0.0`.

**Impact by channel:**
- `powerOut`: stays 0.0 → `launchPointFound` never becomes true (same as CL2) → empty DataPoints, silent
- `Voltage`, `Current`, `Ripple`, `PowerOut`, `Speed`, `Temperature`, etc.: all return 0.0
- `ThrottleCal`: MinMs, NeutralMs, MaxMs all default to hardcoded values; metadata overrides silently fail

**Result for a European-locale user:** A Castle CSV loads as an empty trace, with no error message. Every charge of this label.

**Note:** The `CsvConfiguration(CultureInfo.InvariantCulture)` at line 26 only affects CsvHelper's internal typed reads. Since `GetField<string>` is used everywhere, this setting has no effect on the raw field values. The InvariantCulture setting is misleading — it does not protect the subsequent `double.TryParse` calls.

### CL4 — Debug log `StreamWriter` not in `using` — leaks on exception ⚠️ MEDIUM

**Lines:** 35–53, 241

```csharp
StreamWriter log = null;
if (_configService.IsDebugLoggingEnabled())
{
    try { log = new StreamWriter(logPath, false); ... }
    catch (IOException) { }
}

// ... long try-less code ...

log?.Close();   // only reached if no exception is thrown
```

`log` is a `StreamWriter` that is manually closed at line 241. It is NOT in a `using` statement. If any exception is thrown inside the CSV reading block — CsvHelper exception, field access, etc. — execution jumps to the caller's catch block and line 241 is never reached. The file handle for `debug_log.txt` remains open until the GC finalizer runs.

On Windows, an open file handle prevents other processes from opening the same file with write access. If the user loads another file while the handle is still open (before GC), a second `new StreamWriter(logPath, false)` would fail with an `IOException`, which is caught and silently ignored — meaning **debug logging silently stops working on the second load**.

**Fix:** Wrap `log` in a `using` statement or a `try/finally`.

### CL5 — 10,000 row cap silently truncates long logs ⚠️ LOW

**Line:** 124

```csharp
if (rowIndex > rowMax) { log?.WriteLine("Row limit hit!"); break; }
```

Castle logs rarely exceed 10,000 rows (10,000 × 50ms = 8.3 minutes of data). However, if a user captures a long session (warm-up laps before drag runs), the trim silently drops all data after row 10,000. The user gets a truncated trace and no warning.

The `log?.WriteLine("Row limit hit!")` only writes to the debug log, which is disabled by default. **No user notification is shown.**

### CL6 — Neither "RPM" nor "Speed" column: RPM stays 0 ⚠️ LOW

**Line:** 152

```csharp
string rpmField = csv.HeaderRecord.Contains("RPM") ? "RPM" : "Speed";
```

If neither column exists, `"Speed"` is used. `GetDouble("Speed")` returns 0.0 (MissingFieldFound = null). All RPM values are zero. The RPM channel plots as a flat line at the bottom. No error.

Additionally, `csv.HeaderRecord.Contains("RPM")` does a string equality check (not substring). The column names "RPMs", "RPM (raw)", etc. would fall through to the "Speed" fallback.

### CL7 — Two launch detections can disagree ⚠️ LOW

**Design issue spanning lines 138–149 and 197–215**

`CsvLoader.Load()` performs launch detection twice:

1. **In-loop filter (line 140):** `powerOut >= 5.0` — a simple threshold. Keeps all rows from the first `powerOut >= 5.0` onward. Rows before this are skipped.

2. **Post-loop detection (line 197):** `DetectDragStartIndex()` — a sophisticated debounced algorithm using throttle transition + 5-point sustain window + multi-metric trigger score.

These two detectors can disagree. A car that applies light throttle (powerOut briefly > 5W) before the real launch gets through the in-loop filter, adding those pre-launch rows to `DataPoints`. Then `DetectDragStartIndex` must find the actual launch within those rows. If the pre-launch power blip is long enough, the rows included represent a different event from the actual drag run, and `DetectDragStartIndex` may identify the pre-launch blip as the launch point.

No crash, but the resulting trim window may be anchored to the wrong event.

## RaceBoxLoader.cs — Failure Map: `LoadHeaderOnly`

### RH1 — `int.Parse` without `TryParse` guard on run count ⚠️ HIGH

**Line:** 231

```csharp
// Line 50 (in LoadTelemetry): hasRunCount check uses int.TryParse — safe
// Line 231 (in LoadHeaderOnly): uses int.Parse directly — unsafe
int runCount = int.Parse(allRows[7][1]);
```

The guard at line 224 only checks `allRows[7].Length <= 1` (column exists). It does not verify the value is numeric. If `allRows[7][1]` contains `"N/A"`, `""`, or any non-integer text, `int.Parse` throws `FormatException`.

**Propagation:** All three RaceBox load handlers call `LoadHeaderOnly` with no try/catch. `FormatException` propagates as an unhandled exception in an `async void` handler — app crash. (Audit 04, Findings 1–3.)

**Contrast:** `LoadTelemetry` uses `int.TryParse(allRows[7][1], out _)` at line 50 to validate before `int.Parse` at line 64. `LoadHeaderOnly` omits this safe pattern.

---

### RH2 — Split time `double.TryParse` without CultureInfo ⚠️ HIGH

**Line:** 278

```csharp
var parsed = times
    .Select(t => double.TryParse(t, out var val) ? val : 0)
    .ToList();
bool allNonZero = parsed.All(val => val > 0);
```

RaceBox split times are stored as semicolon-delimited decimal numbers (e.g., `"0.342;1.087;3.421"`). On a German/French locale, `double.TryParse("0.342")` returns `false` (decimal point not valid). Every value becomes 0. `allNonZero` is false for every run row. The loop finds no complete run. `FirstCompleteRunIndex = null`.

**Consequence in callers:** When `FirstCompleteRunIndex` is null, the handlers display a warning and typically default to run index 1. The wrong run's telemetry may be loaded, or a warning is shown. This is handled, but the root cause is the missing CultureInfo.

### RH3 — Duplicate null/empty check for Row 9 ⚠️ LOW

**Lines:** 228–238

```csharp
if (allRows[8].Length <= 1)
    throw new Exception("Row 9 is missing ...");   // first check

// ...

if (allRows[8].Length <= 1 || string.IsNullOrWhiteSpace(allRows[8][1]))
    throw new Exception("Row 9 is missing ...");   // identical check repeated
```

`allRows[8].Length <= 1` is checked twice with identical throw messages. The second check adds `string.IsNullOrWhiteSpace` for the case where the value is whitespace, but this is dead code relative to the first check — the first check already throws before the second can run. Minor code quality issue; the second check's whitespace guard is correct but unreachable via the path where the first guard already passed.

### RH4 — 100-field cap can exclude the RaceBox signature ⚠️ LOW

**Lines:** 196–197

```csharp
if (fieldCount > 100)
    fieldCount = 100;   // only first 100 fields per row are stored
```

The signature check at line 209:
```csharp
if (!string.Join(",", allRows[0]).Contains("RaceBox"))
```
joins only the first 100 fields. If a RaceBox export were ever to place the identifying text after column 100 (not observed in practice, but possible in a new export format), the check would falsely reject the file.

## RaceBoxLoader.cs — Failure Map: `LoadTelemetry`

### RT1 — `return null` for invalid signature: LoadRaceBox1/3 crash ⚠️ HIGH

**Lines:** 56–59

```csharp
if (!hasRunCount || !hasDisciplines)
{
    MessageBox.Show("This file is not a valid RaceBox export...");
    return null;   // ← null returned to caller
}
```

`LoadTelemetry` is called from:
```csharp
var points = await Task.Run(() => loader.LoadTelemetry(path, runIndex));
// no try/catch in LoadRaceBox1 or LoadRaceBox3
```

When `LoadTelemetry` returns null, `points` is null. Then:
```csharp
var startTime = points.First().Time;   // NullReferenceException
```
in `LoadRaceBox1Button_Click` and `LoadRaceBox3Button_Click`. Crash.

`LoadRaceBox2Button_Click` checks `if (points == null || points.Count == 0)` — handles this case correctly.

**Note:** `MessageBox.Show` was already called inside `LoadTelemetry`, so the user sees a dialog. But the subsequent `NullReferenceException` crash in handlers 1 and 3 shows a second unhandled exception dialog (or crashes silently, depending on `Application.SetUnhandledExceptionMode`).

### RT2 — `return points` (empty list) for out-of-bounds telemetry header ⚠️ HIGH

**Lines:** 75–80

```csharp
if (telemetryHeaderRow >= allRows.Count)
{
    MessageBox.Show("Telemetry section missing or incomplete.", ...);
    return points;   // ← points is an empty List<RaceBoxPoint>, not null
}
```

`points` is `new List<RaceBoxPoint>()` — not null, but empty. The caller receives an empty list. Then:
```csharp
var startTime = points.First().Time;   // InvalidOperationException: sequence contains no elements
```
in `LoadRaceBox1/3Button_Click`. Crash.

`LoadRaceBox2Button_Click` checks `if (points == null || points.Count == 0)` — handles this correctly.

**Key difference from RT1:** The return value is an empty list, not null. The null check in LoadRaceBox2 works because `points.Count == 0` is true. LoadRaceBox1/3 have no check at all, so either null or empty list causes a crash there.

### RT3 — `DateTime.Parse` without `CultureInfo` ⚠️ HIGH

**Lines:** 119, 125

```csharp
DateTime baseTime = DateTime.Parse(telemetryRows[0][timeIndex]);   // L119
// ...
DateTime currentTime = DateTime.Parse(row[timeIndex]);              // L125
```

The RaceBox telemetry timestamp format is typically `"14:23:45.678"` (time-of-day with milliseconds). On most European locales, `DateTime.Parse("14:23:45.678")` will succeed because time-of-day parsing is generally culture-neutral. However:

- If the RaceBox format includes a date component (`"2024-01-15 14:23:45.678"`), European locales expecting `"dd.MM.yyyy"` format will throw `FormatException`.
- If the fractional seconds separator (`.`) is locale-dependent in some edge case, this also fails.
- `DateTime.Parse(null)` throws `ArgumentNullException` if the cell is empty (see RT6).

**Fix:** `DateTime.ParseExact(value, "HH:mm:ss.fff", CultureInfo.InvariantCulture)` — or `DateTime.Parse(value, CultureInfo.InvariantCulture)`.

### RT4 — `double.TryParse` without `CultureInfo` zeros speed and G-Force ⚠️ HIGH

**Lines:** 128–129

```csharp
double speedMps = double.TryParse(row[speedIndex], out var s) ? s : 0.0;
double gForceX  = double.TryParse(row[gForceXIndex], out var g) ? g : 0.0;
```

No `CultureInfo.InvariantCulture`. On a non-EN locale, RaceBox values like `"10.547"` (speed in m/s) fail to parse. `speedMps = 0.0`, `gForceX = 0.0` for all points.

**Cascading effect:**
1. `SpeedMph = 0.0` for all points
2. `DetectRaceBoxLaunchIndex` checks `s1 - s0 > 0.2` etc. — all differences are 0 → falls through to fallback return 0
3. Launch detection falls back to the first row as t=0
4. All data trimmed to [-0.5, +2.5] around row 0
5. In `PlotRaceBoxRun`, `Math.Abs(p.Y) > 0.01` filter — all Y values are 0 → all points filtered → `good.Count < 2` → neither Speed nor G-Force trace is plotted
6. **No user notification at any step.** The RaceBox slot appears to load (filename shows, no error), but plots nothing.

### RT5 — `row[timeIndex]` without length guard ⚠️ MEDIUM

**Line:** 125 (and line 119 for index 0)

The telemetry row filter at line 104–108 ensures rows have `r.Length > runColIndex`:
```csharp
.Where(r => r.Length > runColIndex && ...)
```

But it does NOT ensure `r.Length > timeIndex`. If `timeIndex > runColIndex` (depends on column order), a row could satisfy the run-column filter but have fewer columns than `timeIndex`. Then `row[timeIndex]` throws `IndexOutOfRangeException`.

In practice, the "Time" column appears first in RaceBox exports (index 0), and "Run" appears later. So typically `timeIndex < runColIndex`, making this safe in practice. But it's an implicit assumption with no guard.

---

### RT6 — Null cell value passed to `DateTime.Parse` ⚠️ MEDIUM

**Lines:** 40–43, 119, 125

Fields are read as:
```csharp
fields.Add(csv.GetField(i));
```

CsvHelper returns `null` for an empty field when using `GetField(int)` (untyped). If the time column cell is empty in any row, `row[timeIndex]` is null. Then:
```csharp
DateTime.Parse(null);  // throws ArgumentNullException
```

This would crash on any RaceBox file with an empty timestamp cell (malformed row, trailing empty row, export artifact).

### RT7 — No row cap: entire large file loaded into memory ⚠️ LOW

`LoadTelemetry` reads every row of the file into `allRows` with no limit. A 1-hour RaceBox session at 10Hz = 36,000 rows. `allRows` is a `List<string[]>`, each entry having ~20 string fields. Approximate memory: 36,000 × 20 × ~50 bytes = **~36 MB** allocated before any processing. For very long sessions this could be several hundred MB.

Additionally, the file is read completely in BOTH `LoadHeaderOnly` and `LoadTelemetry` — **the file is read twice per RaceBox load**, doubling both I/O and peak memory. `LoadHeaderOnly` only needs the first ~(9 + runCount) rows; everything after that is unnecessary.

## Cross-Cutting Issues

### RX1 — File read twice per RaceBox load ⚠️ LOW

Every `LoadRaceBox*Button_Click` handler calls:
1. `loader.LoadHeaderOnly(path)` — reads entire file
2. `await Task.Run(() => loader.LoadTelemetry(path, ...))` — reads entire file again

For large files this doubles I/O time. The `allRows` list from `LoadHeaderOnly` is discarded, then rebuilt identically in `LoadTelemetry`. A single read-and-parse method (or a `LoadAll(path)` that returns both header and telemetry) would halve file I/O.

---

### RX2 — `MessageBox.Show` called from data layer ⚠️ MEDIUM

Both loaders call `MessageBox.Show` directly from parse methods:

```csharp
// CsvLoader.cs line 202
MessageBox.Show("No drag pass detected...", ...);

// RaceBoxLoader.cs lines 56, 78, 213
MessageBox.Show("This file is not a valid RaceBox export...", ...);
MessageBox.Show("Telemetry section missing or incomplete.", ...);
```

**Two problems:**
1. **Cross-thread call risk:** RaceBox loading is called from `Task.Run()` — a thread pool thread. `MessageBox.Show` should only be called from the UI thread. WinForms marshals this in practice (it works), but it is technically undefined behavior per the WinForms threading model and is flagged by static analyzers.
2. **Testability:** Any test that exercises the load path would block on a dialog box.

The correct pattern is to throw a typed exception (`InvalidRaceBoxFileException`, `NoDragPassException`) and let the UI layer call `MessageBox.Show` in the catch block on the UI thread.

## Input Scenario Matrix

| Input condition | CsvLoader outcome | RaceBoxLoader outcome |
|-----------------|-------------------|-----------------------|
| File does not exist | `FileNotFoundException` — caught by Castle handlers | `FileNotFoundException` — **unhandled** in all 3 RaceBox handlers |
| Empty file | `CsvHelperException` at ReadHeader — caught by Castle handlers | Empty allRows → MessageBox + null/return — caller crash in 1+3 |
| Header only, no data rows | Empty DataPoints, no error shown | Same as empty file |
| Data rows only, no header | CsvHelper reads row 1 as header; mismatched column names; all fields return 0 | N/A (RaceBox uses headerless parsing) |
| Missing "Power-Out" column | **Silent empty DataPoints** — no user notification | N/A |
| Missing "Time" column (RaceBox) | N/A | `throw Exception("Required columns missing")` — unhandled |
| Non-EN locale | All numeric values = 0.0 — silent | Speed/G-Force = 0 — silent; split times fail — no complete run found |
| File truncated mid-row | CsvHelper handles gracefully (incomplete row ignored) | Same |
| 0 valid telemetry rows for run | N/A | `throw Exception("No telemetry rows")` — unhandled |
| > 10,000 rows | Silent truncation | No limit — full file in memory |
| Negative or zero runCount (RaceBox) | N/A | Loop from 0 to runCount-1 doesn't execute — no crash |
| runCount larger than actual rows | Per-row bounds check at L258 — silently skips missing rows | Same |
| Split times with non-numeric values | N/A | `double.TryParse` → 0, row skipped — silent |
| File with CRLF line endings | CsvHelper handles transparently | Same |
| File with BOM | `StreamReader` default handles UTF-8 BOM | Same |

### Immediate (correctness/crash):
1. **RT3** — Add `CultureInfo.InvariantCulture` to `DateTime.Parse` in `LoadTelemetry` (5 min)
2. **CL3** — Add `CultureInfo.InvariantCulture` to all `double.TryParse` calls in `CsvLoader` (10 min)

### Short-term (silent failures with no user feedback):
6. **CL2** — If `DataPoints.Count == 0` after the loop, show a message: `"No data was read. The 'Power-Out' column may be missing or the file format is unexpected."` (10 min)

8. **CL4** — Wrap `log` in `using` or `try/finally` (5 min)
9. **RX2** — Move `MessageBox.Show` out of loaders; throw typed exceptions, catch in UI (1 day — enables testability)
10. **RX1** — Merge `LoadHeaderOnly` + `LoadTelemetry` into a single file-read path, or cache `allRows` between calls (1–2 days — also fixes the double I/O)
