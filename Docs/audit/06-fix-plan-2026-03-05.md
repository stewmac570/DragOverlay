| # | File | Problem | Risk | Fix | Effort | Source |

## Tier P2 — Crash or silent zero on non-standard input

These crash the app or silently zero all data when the input is anything other than an English-locale, well-formed file.

| # | File | Problem | Risk | Fix | Effort | Source |
|---|------|---------|------|-----|--------|--------|
| 6 | `RaceBoxLoader.cs` L119, L125 | `DateTime.Parse` without `CultureInfo.InvariantCulture`. On locales where the date/time format differs from the RaceBox export format (any format that includes a date component), throws `FormatException` — unhandled crash on load. | HIGH | `DateTime.Parse(value, CultureInfo.InvariantCulture)` or `DateTime.ParseExact(value, "HH:mm:ss.fff", CultureInfo.InvariantCulture)` | S | 04 §7, 05 §RT3 |
| 7 | `CsvLoader.cs` L81–95, L135, L159 | `double.TryParse` without `CultureInfo.InvariantCulture` in the metadata section and in `GetDouble()`. On any non-EN locale, Castle values like `"8.4"` fail to parse (`.` is not the decimal separator). Every numeric channel returns 0.0. **The `CsvConfiguration(CultureInfo.InvariantCulture)` setting does NOT help here — it only affects typed CsvHelper reads, not raw `string` field parsing.** | HIGH | Pass `CultureInfo.InvariantCulture, NumberStyles.Float` as the second and third arguments to every `double.TryParse` in both metadata parsing and `GetDouble()`. | S | 05 §CL3, 01 §M2 |
| 8 | `RaceBoxLoader.cs` L128–129 | `double.TryParse` without `CultureInfo` for speed and G-Force values in `LoadTelemetry`. On non-EN locale, all values parse as 0.0. Combined with the near-zero filter in `PlotRaceBoxRun` (PM5), nothing plots and no error is shown. | HIGH | Same `CultureInfo.InvariantCulture` fix as #7. | S | 05 §RT4 |
| 9 | `RaceBoxLoader.cs` L278 | `double.TryParse` without `CultureInfo` for split times in `LoadHeaderOnly`. On non-EN locale, all split times fail; `firstCompleteRunIndex` stays null; callers fall back to run index 1 — may load wrong run. | HIGH | Same `CultureInfo.InvariantCulture` fix as #7. | S | 05 §RH2 |
| 10 | `RaceBoxLoader.cs` L231 | `int.Parse(allRows[7][1])` in `LoadHeaderOnly` with no `TryParse` guard. The value-existence check at L224 only confirms the column is present, not that its value is numeric. A non-integer run count throws `FormatException` — unhandled crash. Note: `LoadTelemetry` correctly uses `int.TryParse` at L50. | HIGH | Replace with `int.TryParse(allRows[7][1], out int runCount) ? runCount : 0` with a check for runCount <= 0. | S | 05 §RH1 |
| 11 | `PlotManager.cs` L669 | Near-zero Y filter `Math.Abs(p.Y) > 0.01` removes RaceBox data points where the G-Force value is at or near zero. G-Force legitimately crosses zero during a drag pass, creating artificial gaps in the trace. The filter is harmless for the Speed channel but harmful for G-Force. | HIGH | Apply the filter only to the Speed channel. For G-Force, use a much smaller guard (1e-9) that only catches true invalid/NaN values. | S | 03 §PM5 |

| # | File | Problem | Risk | Fix | Effort | Source |
|---|------|---------|------|-----|--------|--------|
| 12 | `CsvLoader.cs` | Missing or renamed "Power-Out" column (e.g., `"Power Out"`, `"PowerOut"`) causes every row to be skipped. `runData.DataPoints.Count == 0` at exit. **No MessageBox, no log, no UI indication** — the file appears to load (filename is shown in the button) but the chart is empty. | HIGH | After the parsing loop, if `runData.DataPoints.Count == 0`, show: `"No data was read from this file. The 'Power-Out' column may be missing or renamed.\nExpected column name: 'Power-Out'."` | S | 05 §CL2 |
| 13 | `MainForm.cs` | `OnRpmModeChanged()` builds its own run dictionary containing only `run1`, `run2`, `run3`. When RPM mode changes, RaceBox overlays (runs 4–6) vanish from the chart because they are not passed to `PlotRuns`. | MEDIUM | Replace the hand-built dictionary with `PlotAllRuns()`. The only reason for the separate dict was to apply the RPM scale mid-flight, but `PlotAllRuns()` calls `PlotManager.PlotRuns()` which re-reads `_isFourPoleMode`. | S | 01 §M9 |
| 14 | `PlotManager.cs` L227–231 | `FitToData()` calls `_plot.Refresh()` BEFORE `_plot.Plot.Axes.AutoScale()`. The render uses the pre-autoscale axis limits; the limit change is never rendered. The button does nothing visually. | HIGH | Swap the two lines: `AutoScale()` first, then `Refresh()`. | S | 03 §PM1 |
| 15 | `PlotManager.cs` L950 | `GetTimeRange` only checks `"RaceBox Speed"` for the RaceBox time range. If the speed channel has no valid points (e.g., filtered to zero by the PM5 near-zero issue, or locale issue), the time range falls back to (-1, 10), generating ~2,200 tick objects for an 11-second window. | MEDIUM | Check both `"RaceBox Speed"` and `"RaceBox G-Force X"` channels (or iterate all `run.Data` keys) and take the union time range. | S | 03 §PM10 |
| 16 | `MainForm.cs` | `SyncRunTypeUI()` creates `new ToolTip(this.components)` on every call. Every RunType pill switch leaks a GDI handle. Under repeated toggling, GDI handle exhaustion can cause UI rendering failures (windows stop painting). | MEDIUM | Promote `ToolTip` to a private field, initialised once in the constructor or `InitializeComponent`. | S | 01 §M8 |
| 17 | `RaceBoxLoader.cs` | The entire file is read twice per RaceBox slot load: once in `LoadHeaderOnly` and once in `LoadTelemetry`. `LoadHeaderOnly` only needs the first ~(9 + runCount) rows (~15 rows for a typical file) but reads everything. For large RaceBox files (long sessions), this doubles I/O time. | LOW | Either (a) pass `allRows` from `LoadHeaderOnly` into `LoadTelemetry` to avoid a second read, or (b) have `LoadHeaderOnly` stop reading after the required header rows. | M | 05 §RX1 |

## Tier P4 — Performance: hot path

These affect the user's perceived responsiveness. Impact grows with number of loaded runs.

| # | File | Problem | Risk | Fix | Effort | Source |
|---|------|---------|------|-----|--------|--------|
| 18 | `PlotManager.cs` L800–809 | `FormsPlot_MouseMove` finds the nearest point via `OrderBy(p => Math.Abs(p.X - mouseCoord.X)).First()` — O(n log n) sort on every scatter on every mouse pixel. With 6 runs loaded (~18 scatters × ~600 points each), this is ~81,000 comparisons per pixel. Followed by a second O(n) linear scan to find the index. | HIGH | Replace with a single O(n) linear scan: `int nearestIdx = 0; double minDist = double.MaxValue; for (int j = 0; j < pts.Count; j++) { double d = Math.Abs(pts[j].X - mouseCoord.X); if (d < minDist) { minDist = d; nearestIdx = j; } }` | M | 03 §PM2–PM3 |
| 19 | `PlotManager.cs` L822 | `_plot.Refresh()` called on every `MouseMove` event — full ScottPlot render at mouse movement speed (up to ~200 events/second). With 6 loaded runs, this makes hover sluggish. | HIGH | Add a 60fps cap: `if ((DateTime.UtcNow - _lastRefresh).TotalMilliseconds < 16) return;` at the top of the handler; update `_lastRefresh` before calling `Refresh()`. Add `private DateTime _lastRefresh = DateTime.MinValue;` field. | S | 03 §PM4 |
| 20 | `PlotManager.cs` L827–841 | `GetChannelsWithRaw` returns a tuple with `(Label, RawYs, ScaledYs)` where `ScaledYs` is always an identical copy of `RawYs`. The call site discards `ScaledYs` with `_`. This allocates ~48 KB of duplicate arrays per Castle run per full replot (10 channels × 600 points × 8 bytes × 2). | MEDIUM | Remove the `ScaledYs` element from the tuple entirely. The RPM scaling is applied at the call site anyway. | S | 03 §PM9 |
| 21 | `PlotManager.cs` L904, L399 | `GetTimeRange(runsBySlot)` is called twice per `PlotRuns()` execution — once inside `ApplyManualTimeTicks` and once explicitly 10 lines later. Iterates all run data twice. | LOW | Compute once before calling `ApplyManualTimeTicks`, pass the result in, use it again for the axis limit set. | S | 03 §PM11 |
| 22 | `PlotManager.cs` L909–912 | Tick generation loop uses `pos += 0.05` floating-point accumulation over 60 iterations — drift accumulates. Generates 660 `AddMajor`/`AddMinor` calls per replot (including 60 string allocations). | LOW | Use index-based: `for (int i = 0; ...) { double pos = minX + i * 0.05; ... }` Avoids floating-point drift. Same fix for minor ticks. | S | 03 §PM13 |

| # | File | Problem | Risk | Fix | Effort | Source |

The `debug_log.txt` file handle stays open until GC, preventing the next session from opening the file for writing (silent debug logging failure). | LOW | Change to `using StreamWriter? log = ...` or wrap in `try/finally { log?.Close(); }`. | S | 05 §CL4 |
| 24 | `PlotManager.cs` L417–418 | Exact duplicate comment: `// Made public because MainForm calls this` appears twice consecutively on lines 417 and 418 above `SetupAllAxes()`. | LOW | Delete one. | S | 03 §PM16 |

This survives because `PlotRuns()` clears and rebuilds all rules before any render, but the null rules are a latent correctness risk. | LOW | Either remove the constructor call (axis locks will be applied on first `PlotRuns()`) or call `SetupAllAxes()` first to initialise the axes, then call `ReapplyAxisLocks()`. | S | 03 §PM7 |

On Windows, `File.AppendAllText` with a non-Unicode encoding will corrupt these characters. | LOW | Either use `FileStream` with `Encoding.UTF8` explicitly in `Logger.cs`, or replace emojis with plain-text labels. | S | 01 §L11 |

## Tier P6 — Structural: architectural improvements (planned work, not hotfixes)

These are deliberate refactors that reduce the long-term cost of adding features. Each is a self-contained change that can be done as its own PR.

| # | File | Problem | Risk | Fix | Effort | Source |
|---|------|---------|------|-----|--------|--------|
| 29 | `MainForm.cs` | `LoadRaceBox1Button_Click`, `LoadRaceBox2Button_Click`, and `LoadRaceBox3Button_Click` are ~50-line handlers with nearly identical bodies. The only differences are the slot number (4/5/6), the button reference, and the label. Three copies of the same bug surface means three copies of every future fix. | MEDIUM | Extract a single `LoadRaceBoxSlot(int slot)` method. Pass slot number; derive button reference, label, and run variable from the slot. Eliminates the triplication and ensures the try/catch fix (#1–2) only needs to be written once. | M | 01 §M16, 02 §Risk1 |
| 30 | `Services/CsvLoader.cs`, `Services/RaceBoxLoader.cs` | Both loaders call `MessageBox.Show()` directly. This makes them untestable, creates cross-thread call risk (loaders are called from `Task.Run`), and violates the service/UI layer boundary. | MEDIUM | Have loaders throw typed exceptions (`NoDragPassException`, `InvalidRaceBoxFileException`, `MissingColumnException`). Catch in the load handlers and call `MessageBox.Show` from the UI thread there. Enables unit tests on load logic. | L | 02 §Risk6, 05 §RX2 |

## Recommended session order

### Session A — "No more crashes" (~75 min)
Fix items 1, 2, 3, 4, 5 (P1) then 6, 7, 8, 9, 10, 11 (P2).

Items 6–9 (CultureInfo) are all the same one-line fix in three files. Do them as a batch. After this session: every known crash path is closed.

### Session B — "Silent failures become visible" (~2 hr)
Fix items 12 (empty DataPoints notification), 13 (RPM mode drops RaceBox), 14 (FitToData order), 15 (time range), 16 (ToolTip leak).

After this session: wrong-output scenarios show helpful error messages instead of silent empty plots.

### Session C — "Hover performance" (~2 hr)
Fix items 18 (linear nearest-point scan) and 19 (hover render throttle). These together make hover responsive with 6 loaded runs. Items 20–22 are minor cleanups to bundle with this session.

### Session D — "Structural cleanup" (planned)
Items 29 (shared LoadRaceBoxSlot), 30 (exceptions out of loaders), then the architecture audit's longer-term items: `run1..run6` → `Dictionary<int, RunData>` (Audit 02 Recommendation 4).
