## Detailed Findings

### 🔴 C1 — RaceBoxLoader.cs: Crash when `points` is empty after `LoadTelemetry`

**Location:** `RaceBoxLoader.cs:151–154`

```csharp
int launchIndex = DetectRaceBoxLaunchIndex(points);
double t0 = points[launchIndex].Time.TotalSeconds;  // ← CRASH if points is empty
```

`DetectRaceBoxLaunchIndex` returns `0` as a fallback when no launch is found. If `points` is empty (all telemetry rows failed the `run == selectedRunIndex` filter), then `points[0]` throws `ArgumentOutOfRangeException`. There is no guard before this line.

**Fix:** Add `if (points.Count == 0) return points;` before the launch detection block.

---

### 🔴 C2 — RaceBoxLoader.cs: `DateTime.Parse` without culture

**Location:** `RaceBoxLoader.cs:119, 125`

```csharp
DateTime baseTime = DateTime.Parse(telemetryRows[0][timeIndex]);
DateTime currentTime = DateTime.Parse(row[timeIndex]);
```

These calls use the current OS locale. On a machine where date format is not `M/d/yyyy H:mm:ss` (e.g., a European locale with `dd.MM.yyyy`), these will either parse incorrectly or throw `FormatException`, crashing the entire RaceBox load.

**Fix:** `DateTime.Parse(value, CultureInfo.InvariantCulture)` on both lines.

---

### 🔴 C3 — PlotManager.cs: Y-axis channel switch has no default clause

**Location:** `PlotManager.cs:339–351`

```csharp
s.Axes.YAxis = channelLabel switch
{
    "RPM"          => rpmAxis,
    "Throttle %"   => throttleAxis,
    // ...
    "Acceleration" => accelAxis,
    // ← NO DEFAULT
};
```

C# switch expressions throw `SwitchExpressionException` when no arm matches. If a new channel is added to `GetChannelsWithRaw` but not added to this switch, loading that log crashes. There is currently no `_` fallback.

**Fix:** Add `_ => throttleAxis` (or any safe fallback axis) as the final arm.

---

### 🔴 C4 — MainForm.cs: `LoadRaceBox1Click` and `LoadRaceBox3Click` crash on empty telemetry

**Location:** `MainForm.cs:497–502` (slot 1), `MainForm.cs:710` (slot 3)

```csharp
// LoadRaceBox1Button_Click — no null/empty check
Logger.Log($"📈 RaceBox telemetry parsed: {points.Count} rows for Slot 1");
int firstRecord = points.First().Record;   // ← CRASH if points is empty
int lastRecord  = points.Last().Record;    // ← CRASH if points is empty
```

`LoadRaceBox2Button_Click` has the correct guard (lines 621–626). LoadRaceBox1 and LoadRaceBox3 do not.
For Slot 3 the risk is slightly different: `points.Count` is accessed (line ~710) without a prior null check, then further accesses follow.

**Fix:** Add the same null/empty guard used in LoadRaceBox2:
```csharp
if (points == null || points.Count == 0)
{
    MessageBox.Show("RaceBox telemetry is empty or could not be parsed.", ...);
    return;
}
```

### 🔴 C5 — PlotManager.cs: Full chart `Refresh()` fires on every mouse pixel

**Location:** `PlotManager.cs:772–823` (`FormsPlot_MouseMove`)

Every mouse movement over the chart triggers:
1. A linear scan through all scatters
2. An O(n log n) `OrderBy` on each scatter's data points (up to 600 pts × 30 scatters = 18 000 comparisons)
3. `_plot.Refresh()` — a full GPU-backed chart redraw

With 3 runs × 10 channels loaded, this will cause visible lag and stuttering as the mouse moves. On lower-end hardware it becomes a slider.

**Two separate sub-issues:**
- The `OrderBy().First()` at line 800 is O(n log n) per scatter. Replace with a simple linear `min` scan for O(n).
- `_plot.Refresh()` should be throttled (e.g., only refresh if the cursor X moved by more than 1 pixel, or use a timer to cap refresh rate at ~60 Hz).

### 🟡 M1 — Logger.cs: Log file grows unbounded

**Location:** `Logger.cs:24–25`

```csharp
//File.WriteAllText(_logPath, $"=== App started: {DateTime.Now} ===\n");
```

The startup truncation is commented out. `File.AppendAllText` accumulates indefinitely. Over many sessions this file can grow to tens of MB in the AppData folder.

### 🟡 M2 — CsvLoader.cs: `double.TryParse` ignores culture

**Location:** `CsvLoader.cs:158`

```csharp
return double.TryParse(raw, out double result) ? result : 0.0;
```

`double.TryParse` with no `CultureInfo` argument uses `Thread.CurrentThread.CurrentCulture`. On a system where the decimal separator is `,` (common across Europe), values like `"1.500"` parse to 0. The file reader is set to `InvariantCulture` for CSV structure, but field values use the overloaded `TryParse`.

**Fix:** `double.TryParse(raw, NumberStyles.Any, CultureInfo.InvariantCulture, out double result)`

### 🟡 M3 — CsvLoader.cs: Silent row truncation at 10 000 rows

**Location:** `CsvLoader.cs:119, 124–127`

```csharp
const int rowMax = 10000;
if (rowIndex > rowMax)
{
    log?.WriteLine("Row limit hit!");
    break;
}
```

The break is silent from the user's perspective — no `MessageBox`, no indication in the UI. For a typical 3-second drag pass (60 rows) this is irrelevant, but for speed runs or long logs the user may load a file and not realise it was truncated.

**Fix:** Set `runData.WasTruncated = true` and show a warning in the calling code, or at minimum log a visible warning via `Logger.Log`.

### 🟡 M4 — CsvLoader.cs: Fixed 50 ms timestamp reconstruction

**Location:** `CsvLoader.cs:167`

```csharp
Time = rowIndex * 0.05,
```

The real timestamp from the CSV is ignored. Time is reconstructed from the row counter × a hardcoded 50 ms step. If Castle ever changes logging frequency, or the log contains non-uniform intervals, the time axis will be wrong. There is no verification that the CSV's time column matches this assumption.

**Fix:** Parse the actual time column from the CSV (if present) and use it. Fall back to `rowIndex * 0.05` if the column is absent or invalid.

### 🟡 M5 — PlotManager.cs: O(n log n) sort per scatter per mouse move

**Location:** `PlotManager.cs:800`

```csharp
var nearest = pts.OrderBy(p => Math.Abs(p.X - mouseCoord.X)).First();
```

This creates an array, sorts it, and discards everything except the first element — for every scatter, on every `MouseMove` event. With 30 scatters × 300 pts = 9 000 comparisons just for the sort, this adds up.

**Fix:** Replace with a linear scan:
```csharp
var nearest = pts[0];
double minDist = Math.Abs(pts[0].X - mouseCoord.X);
foreach (var p in pts)
{
    double d = Math.Abs(p.X - mouseCoord.X);
    if (d < minDist) { minDist = d; nearest = p; }
}
```

### 🟡 M6 — PlotManager.cs + MainForm.cs: Duplicated `_isFourPoleMode` state

**Locations:** `PlotManager.cs:64`, `MainForm.cs:40`

Both classes maintain an independent `_isFourPoleMode` boolean. They are kept in sync by calling `_plotManager.SetFourPoleMode(...)` from MainForm, but any code path that forgets to sync them will produce a chart that disagrees with hover values, or vice versa.

**Fix:** Remove the field from MainForm entirely; add `PlotManager.IsFourPoleMode { get; }` and derive the value from there when needed.

### 🟡 M7 — PlotManager.cs: `SetSpeedMode` has commented-out side effects

**Location:** `PlotManager.cs:106–111`

```csharp
public void SetSpeedMode(bool isSpeedRun)
{
    _activeProfile = isSpeedRun ? ScaleProfile.SpeedRun : ScaleProfile.Drag;
    //ReapplyAxisLocks();   ← commented out
    //_plot.Refresh();      ← commented out
}
```

Calling `SetSpeedMode` alone does nothing visible. Every caller also calls `SetupAllAxes()` and `RefreshPlot()` manually afterwards, creating an implicit API contract that is easy to violate. If a future call site omits those steps, the profile switches but the chart does not update.

**Fix:** Either uncomment the two lines inside `SetSpeedMode` so it is self-contained, or remove `SetSpeedMode` as a standalone API and fold the profile into a parameter of `SetupAllAxes`.

### 🟡 M8 — MainForm.cs: `ToolTip` leak in `SyncRunTypeUI`

**Location:** `MainForm.cs:1244`

```csharp
var tt = new ToolTip(this.components);
```

A new `ToolTip` is created every time `SyncRunTypeUI` is called. This is called from `UpdateRunTypeLockState()`, which is called after every run load and delete. Each `ToolTip` registers with `IContainer components`, accumulating GDI handles over the session.

**Fix:** Declare `_runTypeTip` as a field, create it once in the constructor, and call `SetToolTip` on the existing instance.

### 🟡 M9 — MainForm.cs: `OnRpmModeChanged` omits RaceBox runs from replot

**Location:** `MainForm.cs:969–982`

```csharp
private void OnRpmModeChanged(bool isFourPole)
{
    // ...
    var runsToPlot = new Dictionary<int, RunData>();
    if (run1 != null) runsToPlot[1] = run1;
    if (run2 != null) runsToPlot[2] = run2;
    if (run3 != null) runsToPlot[3] = run3;
    // run4, run5, run6 MISSING
    _plotManager.PlotRuns(runsToPlot);
```

When the user switches RPM pole mode, RaceBox overlays (slots 4–6) are wiped from the chart because they are not included in the `PlotRuns` call.

**Fix:** Replace the local dictionary construction with `PlotAllRuns()`, which already collects all six slots.

### 🟡 M11 — MainForm.cs: Triple redundant calls after run load

**Location:** `MainForm.cs:319–323` (and identically in LoadRun2, LoadRun3)

```csharp
PlotAllRuns();
_plotManager.SetSpeedMode(_isSpeedRunMode);   // ← sets a flag only (side effects commented out)
_plotManager.SetupAllAxes();                  // ← sets up axes (already done by PlotAllRuns → PlotRuns)
_plotManager.RefreshPlot();                   // ← extra Refresh after PlotAllRuns already refreshed
```

`PlotAllRuns()` → `PlotRuns()` already calls `ReapplyAxisLocks()` and `_plot.Refresh()`. The three subsequent calls are largely redundant. `SetupAllAxes` has `null` guards so it skips axis creation, but it still iterates and re-removes/re-adds `LockedVertical` rules twice per load.

**Fix:** Remove the three lines after `PlotAllRuns()` in the load handlers; or encapsulate mode + axes + refresh into a single `ApplyAndRefresh()` method.

---

### 🟡 M12 — ConfigService.cs: Bare `catch {}` on config load

**Location:** `ConfigService.cs:52`

```csharp
catch
{
    _config = new Config();
    Save();
}
```

This catches everything: `JsonException` (corruption, expected), `UnauthorizedAccessException` (permissions issue, not expected), `OutOfMemoryException`, etc. If the file exists but can't be read due to permissions, `Save()` then runs, which will also likely fail and throw — but that exception propagates unhandled from the constructor.

**Fix:** `catch (JsonException)` for the corruption case. Let genuine I/O errors propagate so the user sees a meaningful error at startup.

---

### 🟡 M13 — RaceBoxLoader.cs: Entire file loaded into memory

**Location:** `RaceBoxLoader.cs:33–44` and `189–203`

Both `LoadTelemetry` and `LoadHeaderOnly` read the complete file into `List<string[]>` before any processing. RaceBox exports can span multiple sessions and grow large (thousands of telemetry rows). A streaming approach would reduce peak memory.

**Fix (medium-term):** Stream through the file once, switch modes when the telemetry header is found, and stop after the desired run's rows are consumed.

### 🟡 M15 — ChannelToggleBar.cs: `Font` objects never disposed

**Location:** `ChannelToggleBar.cs:182`, `MainForm.cs:1225, 1231`

```csharp
Font = new Font(SystemFonts.DefaultFont.FontFamily, 10, FontStyle.Bold)
```
```csharp
runTypeDrag.Font = new Font(runTypeDrag.Font, speed ? FontStyle.Regular : FontStyle.Bold);
```

### 🟡 M16 — MainForm.cs: LoadRaceBox1 and LoadRaceBox3 lack full try/catch

**Location:** `MainForm.cs:462–592` (slot 1), `MainForm.cs:684–764` (slot 3)

`LoadRaceBox1Button_Click` and `LoadRaceBox3Button_Click` are `async void` but have no try/catch wrapping the `await` call. An exception thrown inside `LoadTelemetry` (after the `await`) propagates to the SynchronizationContext and crashes the application with no user-facing message. `LoadRaceBox2Button_Click` (slot 2) has the correct guard.

**Fix:** Wrap the entire handler body in `try { ... } catch (Exception ex) { MessageBox.Show(...); Logger.Log(...); }`.

### 🟡 M17 — MainForm.cs: LoadRaceBox1/2/3 are 90% identical

**Location:** `MainForm.cs:462–764`

All three RaceBox load handlers share ~80 lines of identical code: loading header, checking null, awaiting telemetry, building `RunData`, registering the run, updating the toggle bar, enabling buttons. This is copy-paste code. A future change (e.g., adding a new RaceBox channel) must be made in three places.

**Fix:** Extract to `private async Task LoadRaceBoxSlot(int uiSlot, int plotSlot, Button loadBtn, Button toggleBtn, Button deleteBtn)`. Call it from three thin wrappers.

### 🟢 L1 — Logger.cs: `File.AppendAllText` per message is I/O-intensive

**Location:** `Logger.cs:33`

When debug logging is enabled, every `Logger.Log()` call opens the file, writes a line, and closes it. In a busy session this can be hundreds of file open/close operations per second. No impact when disabled (the guard at line 28 short-circuits).

**Fix (if performance matters during debug):** Buffer messages in a `Queue<string>` and flush on a timer, or use `StreamWriter` held open for the session.

### 🟢 L2 — CsvLoader.cs: Fragile colon split for metadata

### 🟢 L9 — RunData.cs: Dual-purpose model is confusing

**Location:** `RunData.cs`

`RunData` serves two incompatible shapes:
- **Castle runs:** use `DataPoints` (flat `List<DataPoint>`)
- **RaceBox runs:** use `Data` (dictionary of channel → `List<DataPoint>`), plus a dummy `DataPoints` populated with time-only entries

The `IsRaceBox` flag is the only discriminator. Code that accesses `run.DataPoints` on a RaceBox slot gets time-only dummy data. This has already caused at least one log statement to read from the wrong property.

**Fix (long-term):** Split into `CastleRunData` and `RaceBoxRunData` inheriting from a `RunData` base, or use a sealed union type.

### 🟢 L10 — GovGain channel: documented but not implemented

**Locations:** README.md, FEATURES.md

`GovGain` is listed as a supported channel in the README and feature docs, but:
- No `govGainAxis` exists in `PlotManager`
- `GetChannelsWithRaw` does not yield it
- It is not in `Config.ChannelVisibility`
- It is not in `ChannelColorMap`

Either the README should be updated to remove it, or the channel should be implemented.

### 🟢 L11 — RaceBoxLoader.cs: Magic row index numbers

**Location:** `RaceBoxLoader.cs:50, 51, 64, 68, 254–258`

Row offsets `7`, `8`, `9` (for run count, disciplines, and run data) appear as literals throughout both `LoadTelemetry` and `LoadHeaderOnly`. If RaceBox changes their export format, these must be found and updated across multiple locations.

**Fix:** Define `const int RB_ROW_RUN_COUNT = 7; const int RB_ROW_DISCIPLINES = 8; const int RB_ROW_DATA_START = 9;`

### 🟢 L12 — PlotManager.cs: `GetTimeRange` ignores `RaceBox G-Force X`

**Location:** `PlotManager.cs:950`

```csharp
if (run.Data.TryGetValue("RaceBox Speed", out var rb) && rb.Count > 0)
```

Only the `RaceBox Speed` channel is used to calculate the X-axis range for RaceBox runs. If a RaceBox run has Speed data but the Speed channel is hidden (or empty), and G-Force X data exists with a wider time range, the X-axis will be clipped incorrectly.
